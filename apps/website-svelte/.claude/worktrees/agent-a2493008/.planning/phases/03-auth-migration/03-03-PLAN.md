**src/lib/lists/convexListStore.svelte.ts:**

Create `createConvexListStore()` that returns a `ListStore` implementation backed by Convex reactive queries and mutations.

Key design decisions (from Research — Pitfall 1 and Pitfall 2):

- The `ListStore` interface has sync return types (`createList` returns `GuestList`). ConvexListStore uses **optimistic local state** for mutating methods: return a placeholder immediately, fire the mutation in background, and let `useQuery()` subscriptions reconcile the real state.
- Read-only properties (`lists`, `activeListItems`) come from `useQuery()` reactive subscriptions, mapped to `GuestList`/`GuestListItem` shapes.

```typescript
import { useQuery, useConvexClient } from "convex-svelte";
import { api } from "../../convex/_generated/api";
import type {
  ListStore,
  GuestList,
  GuestListItem,
  CatalogReference,
  ItemCondition,
} from "./listTypes";
import { SvelteSet } from "svelte/reactivity";
import { catalogKeyForReference } from "./listTypes";
import { toast } from "svelte-sonner";

export function createConvexListStore(): ListStore {
  const client = useConvexClient();

  // Reactive query for all user's lists
  const listsQuery = useQuery(api.authed.lists.listByOwner, {});

  let activeListId = $state<string | null>(null);

  // Reactive query for active list items — skip when no active list
  const itemsQuery = useQuery(api.authed.listItems.byList, () =>
    activeListId ? { listId: activeListId as any } : "skip",
  );

  // Map Convex list doc to GuestList shape
  function mapList(doc: any): GuestList {
    return {
      id: doc._id,
      name: doc.name,
      color: doc.color,
      createdAt: doc._creationTime,
    };
  }

  // Map Convex listItem doc (with catalogType/catalogNumber from byList query) to GuestListItem shape
  function mapItem(doc: any): GuestListItem {
    return {
      id: doc._id,
      listId: doc.listId,
      catalogRef: { type: doc.catalogType, number: doc.catalogNumber },
      condition: doc.condition as ItemCondition,
      quantity: doc.quantity,
      complete: doc.complete,
      tags: doc.tags,
    };
  }

  // Auto-set activeListId when lists load and nothing is selected
  $effect(() => {
    const lists = listsQuery.data;
    if (lists && lists.length > 0 && !activeListId) {
      activeListId = lists[0]._id;
    }
  });

  return {
    get lists() {
      return (listsQuery.data ?? []).map(mapList);
    },
    get activeListId() {
      return activeListId;
    },
    get activeListItems() {
      return (itemsQuery.data ?? []).map(mapItem);
    },

    createList(name: string, color: string): GuestList {
      // Optimistic: create placeholder, fire mutation in background
      const tempId = crypto.randomUUID();
      const placeholder: GuestList = { id: tempId, name, color, createdAt: Date.now() };

      client
        .mutation(api.authed.lists.create, { name, color })
        .then((id) => {
          // Set active to the real Convex ID once mutation completes
          activeListId = id;
        })
        .catch(() => {
          toast.error("You can create up to 3 lists.");
        });

      return placeholder;
    },

    updateList(id, updates) {
      client
        .mutation(api.authed.lists.update, {
          id: id as any, // Convex Id<'lists'>
          ...(updates.name !== undefined ? { name: updates.name } : {}),
          ...(updates.color !== undefined ? { color: updates.color } : {}),
        })
        .catch(() => toast.error("Failed to update list"));
    },

    deleteList(id) {
      if (activeListId === id) {
        const remaining = (listsQuery.data ?? []).filter((l) => l._id !== id);
        activeListId = remaining[0]?._id ?? null;
      }
      client
        .mutation(api.authed.lists.remove, { id: id as any })
        .catch(() => toast.error("Failed to delete list"));
    },

    setActiveList(id) {
      activeListId = id;
    },

    addCatalogItems(listId, refs) {
      // For ConvexListStore, we need catalogItemIds. Resolve type:number -> catalogItemId.
      // The add mutation expects catalogItemId, but we receive CatalogReference (type:number).
      // We need to look up each ref. Use a batch approach via the mutation.
      // However, the authed listItems.add expects catalogItemId (Id<'catalogItems'>), not natural keys.
      // Solution: Add a helper mutation that accepts natural keys, OR resolve on client.
      // For now, use the existing add mutation. The caller must have catalogItemIds.
      // In practice, catalog pages have the Convex doc with _id available.
      // For the ListStore interface compat, we'll fire the mutation with the refs.
      // BUT: CatalogReference only has type+number, not catalogItemId.
      // We need a different approach: add a query to resolve natural keys,
      // or change the add mutation to accept natural keys.
      // PRAGMATIC: The catalog UI components already have the catalogItem _id.
      // We'll store a lookup of type:number -> catalogItemId from the items query.
      // For bulk add from catalog, the calling code should pass items with IDs.
      // For interface compat: fire a helper that resolves keys server-side.

      // Use a simple approach: fire the mutation with natural keys, let server resolve.
      // Actually, the listItems.add mutation takes catalogItemId. We need to either:
      // 1. Change the mutation to accept natural keys (breaks the clean API)
      // 2. Add a resolveAndAdd mutation
      // 3. Accept that addCatalogItems in ConvexListStore expects the ref to be
      //    augmented with catalogItemId at the call site.

      // Going with approach: the ConvexListStore addCatalogItems creates items with
      // default condition/quantity/complete and fires the add mutation.
      // The CatalogReference needs a catalogItemId. In the Convex context,
      // the catalog item IDs are available from the catalog queries.
      // We'll extend this to accept an optional catalogItemId on CatalogReference.
      // For now, log a warning if no catalogItemId available and skip.

      // PRACTICAL: In the actual app, addCatalogItems is called from catalog pages
      // where we have the full Convex doc. We'll expect the callers to pass
      // refs that include the catalogItemId on a custom property.
      // Store the catalogItemId in a Map keyed by type:number from existing items.

      // Simplest working approach: skip items without catalogItemId
      // The catalog UI already provides Convex IDs through its data flow
      toast.info("Adding items...");
      const itemsToAdd = refs
        .map((ref) => ({
          catalogItemId: (ref as any)._id as string,
          condition: "mint" as const,
          quantity: 1,
          complete: true,
          tags: [] as string[],
        }))
        .filter((item) => item.catalogItemId);

      if (itemsToAdd.length > 0) {
        client
          .mutation(api.authed.listItems.add, {
            listId: listId as any,
            items: itemsToAdd as any,
          })
          .catch(() => toast.error("Failed to add items"));
      }
    },

    removeItems(itemIds) {
      client
        .mutation(api.authed.listItems.remove, {
          itemIds: itemIds as any,
        })
        .catch(() => toast.error("Failed to remove items"));
    },

    updateItems(itemIds, updates) {
      client
        .mutation(api.authed.listItems.update, {
          itemIds: itemIds as any,
          updates: {
            ...(updates.condition !== undefined ? { condition: updates.condition } : {}),
            ...(updates.quantity !== undefined ? { quantity: updates.quantity } : {}),
            ...(updates.complete !== undefined ? { complete: updates.complete } : {}),
          },
        })
        .catch(() => toast.error("Failed to update items"));
    },

    duplicateItems(itemIds) {
      // Optimistic: return empty array, let useQuery reconcile
      client
        .mutation(api.authed.listItems.duplicate, {
          itemIds: itemIds as any,
        })
        .catch(() => toast.error("Failed to duplicate items"));
      return [];
    },

    getCompletionPercent(listId) {
      const listItems = (itemsQuery.data ?? []).filter((i) => i.listId === listId);
      if (listItems.length === 0) return 0;
      const completeCount = listItems.filter((i) => i.complete).length;
      return Math.round((completeCount / listItems.length) * 100);
    },

    getOwnedCatalogKeys(listId) {
      const listItems = (itemsQuery.data ?? []).filter((i) => i.listId === listId);
      return new SvelteSet(listItems.map((i) => `${i.catalogType}:${i.catalogNumber}`));
    },

    getItemCount(listId) {
      return (itemsQuery.data ?? []).filter((i) => i.listId === listId).length;
    },

    getOwnedCountByType(listId, type) {
      return (itemsQuery.data ?? []).filter((i) => i.listId === listId && i.catalogType === type)
        .length;
    },

    getOwnedCountByRange(listId, type, start, end) {
      return (itemsQuery.data ?? []).filter(
        (i) =>
          i.listId === listId &&
          i.catalogType === type &&
          i.catalogNumber >= start &&
          i.catalogNumber <= end,
      ).length;
    },
  };
}
```

```typescript
import { toast } from "svelte-sonner";
import type { ConvexClient } from "convex/browser";
import { api } from "../../convex/_generated/api";

const STORAGE_KEY = "diddl-guest-data";
const SESSION_KEY = "diddl-guest-session-id";

interface StoragePayload {
  lists: Array<{ id: string; name: string; color: string; createdAt: number }>;
  items: Array<{
    id: string;
    listId: string;
    catalogRef: { type: string; number: number };
    condition: string;
    quantity: number;
    complete: boolean;
    tags: string[];
  }>;
  activeListId: string | null;
}

/**
 * Get or create a persistent guest session ID for idempotency tracking.
 */
export function getGuestSessionId(): string {
  let id = localStorage.getItem(SESSION_KEY);
  if (!id) {
    id = crypto.randomUUID();
    localStorage.setItem(SESSION_KEY, id);
  }
  return id;
}

/**
 * Check if there is guest data in localStorage that needs migration.
 */
export function hasGuestData(): boolean {
  const raw = localStorage.getItem(STORAGE_KEY);
  if (!raw) return false;
  try {
    const data = JSON.parse(raw) as StoragePayload;
    return data.lists.length > 0;
  } catch {
    return false;
  }
}

/**
 * Migrate guest data from localStorage to Convex.
 * Retries up to 3 times with exponential backoff (1s, 2s, 4s).
 * Returns true on success, false on final failure.
 * localStorage is cleared ONLY on confirmed success (per CF-03).
 */
export async function migrateGuestData(client: ConvexClient): Promise<boolean> {
  const raw = localStorage.getItem(STORAGE_KEY);
  if (!raw) return true; // Nothing to migrate

  let guestData: StoragePayload;
  try {
    guestData = JSON.parse(raw);
  } catch {
    return true; // Corrupt data, nothing to migrate
  }

  if (guestData.lists.length === 0) return true;

  const guestSessionId = getGuestSessionId();
  const delays = [1000, 2000, 4000];

  for (let attempt = 0; attempt <= delays.length; attempt++) {
    try {
      await client.action(api.authed.migration.migrateGuestData, {
        guestSessionId,
        lists: guestData.lists.map((l) => ({
          id: l.id,
          name: l.name,
          color: l.color,
        })),
        items: guestData.items.map((i) => ({
          listId: i.listId,
          type: i.catalogRef.type,
          number: i.catalogRef.number,
          condition: i.condition as any,
          quantity: i.quantity,
          complete: i.complete,
          tags: i.tags,
        })),
      });

      // Success — clear localStorage (per CF-03: only after confirmed success)
      localStorage.removeItem(STORAGE_KEY);
      localStorage.removeItem(SESSION_KEY);
      toast.success("Your collection has been saved to your account.");
      return true;
    } catch (error) {
      if (attempt < delays.length) {
        toast.info("Could not save collection. Retrying...");
        await new Promise((r) => setTimeout(r, delays[attempt]));
      }
    }
  }

  // All retries failed (per D-10)
  toast.warning("Data safe in browser. We'll try again next time.");
  return false;
}
```

**Problem (Pitfall 2):** Svelte `createContext()` sets a value once during component initialization. We cannot reactively swap the store after mount. The solution is to make `listStoreContext` hold a reactive wrapper that delegates to the current underlying store.

**src/lib/lists/listStoreContext.svelte.ts — UPDATE:**

Replace the current implementation with a reactive wrapper approach. Instead of setting the context to a `ListStore` directly, set it to a reactive proxy object that has an internal `$state<ListStore>` delegate. All property accesses and method calls are forwarded to the current delegate.

```typescript
import { createContext } from "svelte";
import type { ListStore } from "./listTypes";

interface ReactiveListStoreHolder {
  readonly store: ListStore;
  setStore(store: ListStore): void;
}

const [internalGetContext, setInternalContext] = createContext<ReactiveListStoreHolder>();

export function getListStoreContext(): ListStore {
  const holder = internalGetContext();
  if (!holder) {
    throw new Error("ListStore context not found");
  }
  return holder.store;
}

export function initListStoreContext(initialStore: ListStore): ReactiveListStoreHolder {
  let currentStore = $state<ListStore>(initialStore);

  const holder: ReactiveListStoreHolder = {
    get store() {
      return currentStore;
    },
    setStore(store: ListStore) {
      currentStore = store;
    },
  };

  setInternalContext(holder);
  return holder;
}
```

IMPORTANT: `getListStoreContext()` still returns `ListStore` so ALL existing consumer components remain untouched. The only change is at the creation site (app layout) which now uses `initListStoreContext` instead of `setListStoreContext`, and gets back a `holder` with `setStore()` for swapping.

**NOTE:** Since `getListStoreContext` returns `holder.store` which is a `$state` value, Svelte 5's reactivity will track it. When `setStore()` is called with a new store, any component that called `getListStoreContext()` and is reading `.lists` or other properties will re-render because the underlying `$state` changed.

Actually, there's a subtlety: `getListStoreContext()` is called once per component and returns the `ListStore` object. If the holder.store changes, existing references won't update. The fix: make `getListStoreContext()` return a **proxy** that always delegates to `holder.store`:

```typescript
export function getListStoreContext(): ListStore {
  const holder = internalGetContext();
  if (!holder) {
    throw new Error("ListStore context not found");
  }
  // Return a proxy that always delegates to the current store
  // This ensures reactivity when the underlying store is swapped
  return new Proxy({} as ListStore, {
    get(_target, prop) {
      const store = holder.store;
      const value = store[prop as keyof ListStore];
      if (typeof value === "function") {
        return value.bind(store);
      }
      return value;
    },
  });
}
```

This Proxy approach ensures that every property access on the returned ListStore goes through `holder.store`, which is reactive via `$state`. When `setStore()` swaps the underlying store, subsequent reads from any component will hit the new store.
